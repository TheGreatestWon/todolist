## 1. 페르소나 정의 (User Personas)

### 페르소나 1: 김미영 (프리랜서 디자이너, 29세)

**배경:**
- 여러 클라이언트의 프로젝트를 동시에 진행하는 프리랜서 디자이너
- 스마트폰과 노트북을 오가며 작업
- 하루에 10-15개의 크고 작은 작업 수행
- 마감일이 중요한 업무 특성

**목표:**
- 각 프로젝트별 작업 마감일을 놓치지 않기
- 이동 중에도 모바일로 할 일 확인 및 추가
- 오늘 우선적으로 해야 할 일을 한눈에 파악
- 완료한 작업을 통해 성취감 느끼기

**니즈:**
- 마감일 기반 자동 정렬로 긴급한 작업 우선 표시
- 모바일에서 빠르게 할 일 추가
- 기한 경과된 작업 명확히 구분
- 간단하고 직관적인 인터페이스

**페인포인트:**
- 복잡한 할 일 관리 도구는 사용법 익히는 데 시간 소요
- 카테고리나 태그 설정이 번거로움
- 여러 앱을 오가며 작업 관리하는 것이 비효율적
- 마감일 지난 작업을 놓치고 클라이언트에게 연락받은 경험

**사용 패턴:**
- **아침(7-9시):** 모바일로 오늘 할 일 확인
- **오전(10-12시):** 데스크탑에서 작업하며 완료 처리
- **오후(2-6시):** 클라이언트 미팅 후 새로운 할 일 추가
- **저녁(9-10시):** 모바일로 내일 계획 수립 및 마감일 조정

---

### 페르소나 2: 박준호 (대학생, 23세)

**배경:**
- 컴퓨터공학 전공 3학년 대학생
- 과제, 팀 프로젝트, 아르바이트 병행
- 주로 노트북 사용, 이동 시 스마트폰 활용
- 체계적인 시간 관리 필요성을 느끼기 시작

**목표:**
- 과제 마감일을 체계적으로 관리
- 아르바이트 스케줄과 학업 병행
- 완료한 과제 목록으로 학기 말 성취감 확인
- 간단하고 무료인 도구로 시작

**니즈:**
- 마감일 중심의 명확한 우선순위
- 복잡한 기능 없이 할 일만 집중 관리
- 데스크탑과 모바일 모두에서 접근
- 별도 앱 설치 없이 브라우저에서 사용

**페인포인트:**
- 종이 메모는 자주 잃어버림
- 여러 과목의 과제 마감일을 기억하기 어려움
- 복잡한 프로젝트 관리 도구는 과한 느낌
- 할 일을 적어놓고도 확인을 안 해서 놓침

**사용 패턴:**
- **아침(8-9시):** 등교 전 모바일로 오늘 할 일 확인
- **점심시간(12-1시):** 스마트폰으로 새로운 과제 추가
- **저녁(7-11시):** 노트북으로 과제하며 완료 처리
- **주말:** 다음 주 과제 마감일 정리

---

### 페르소나 3: 이성은 (직장인, 35세)

**배경:**
- IT 기업 프로젝트 매니저
- 업무와 개인 생활 모두 체계적으로 관리하려는 성향
- 주로 데스크탑 근무, 출퇴근 시 모바일 활용
- 일과 삶의 균형을 중요하게 생각

**목표:**
- 업무 관련 할 일과 개인 할 일을 한곳에서 관리
- 기한 내 작업 완료로 업무 효율성 향상
- 퇴근 후 개인 시간 확보를 위한 시간 관리
- 데이터 보안이 중요 (개인 정보 보호)

**니즈:**
- 회사 업무와 개인 일정을 통합 관리
- 마감일 기반 우선순위로 업무 집중도 향상
- 개인 계정으로 데이터 완전 격리
- 간단하지만 신뢰할 수 있는 도구

**페인포인트:**
- 회사 협업 도구는 개인 일정에 부적합
- 종이 다이어리는 휴대와 수정이 불편
- 복잡한 기능이 많은 도구는 오히려 시간 낭비
- 할 일을 적어도 우선순위 판단이 어려움

**사용 패턴:**
- **출근길(8-9시):** 지하철에서 모바일로 오늘 할 일 확인
- **업무 시간(9-6시):** 데스크탑에서 할 일 추가/완료 처리
- **퇴근길(6-7시):** 모바일로 개인 할 일 추가
- **주말:** 다음 주 계획 수립 및 완료 목록 정리

## 2. 사용자 여정 맵 (User Journey Map)

### 2.1 첫 방문 및 가입 단계

**단계:** 서비스 발견
- **행동:** 검색 또는 지인 추천으로 서비스 접속
- **생각:** "또 다른 할 일 관리 앱인가? 이번엔 간단했으면 좋겠는데..."
- **감정:** 회의적, 조심스러운 기대감
- **터치포인트:** 랜딩 페이지, 로그인 화면
- **기회:** 명확한 가치 제안과 간단한 UI로 첫인상 개선

**단계:** 회원가입 결정
- **행동:** 회원가입 페이지 진입, 이메일과 비밀번호 입력
- **생각:** "가입 절차가 간단하네. 이메일만 있으면 되는구나."
- **감정:** 긍정적, 편안함
- **터치포인트:** 회원가입 폼
- **페인포인트:** 복잡한 필수 정보 요구 시 이탈 가능성
- **기회:** 최소한의 정보만 요구하여 가입 장벽 낮춤

**단계:** 첫 로그인
- **행동:** 가입 완료 후 로그인
- **생각:** "로그인하면 뭐가 나올까? 튜토리얼이 길면 귀찮은데..."
- **감정:** 기대감, 약간의 불안
- **터치포인트:** 로그인 페이지, 할 일 목록 메인 페이지
- **기회:** 즉시 사용 가능한 간단한 인터페이스 제공

---

### 2.2 첫 할 일 생성 단계

**단계:** 빈 화면 마주하기
- **행동:** 할 일 목록 페이지에서 빈 목록 확인
- **생각:** "아무것도 없네. 어디서 추가하지?"
- **감정:** 약간의 혼란, 탐색 욕구
- **터치포인트:** 빈 할 일 목록, 할 일 추가 폼
- **기회:** 명확한 CTA(Call-to-Action)와 안내 메시지

**단계:** 첫 할 일 입력
- **행동:** 제목 입력 필드 클릭, "보고서 작성" 입력
- **생각:** "제목만 입력하면 되나? 설명이나 마감일은?"
- **감정:** 탐색적, 호기심
- **터치포인트:** 할 일 추가 폼 (제목, 설명, 마감일)
- **기회:** 직관적인 폼 레이아웃과 선택적 필드 표시

**단계:** 마감일 설정
- **행동:** 마감일 선택기 클릭, 3일 후 날짜 선택
- **생각:** "날짜 선택이 편하네. 시간은 필요 없나 보다."
- **감정:** 만족, 이해
- **터치포인트:** 날짜 선택기
- **기회:** 직관적인 날짜 선택 UI

**단계:** 첫 할 일 완료
- **행동:** 추가 버튼 클릭
- **생각:** "오, 바로 목록에 나타났네!"
- **감정:** 성취감, 긍정적 놀라움
- **터치포인트:** 할 일 목록에 새 항목 표시
- **기회:** 즉각적인 피드백으로 긍정적 경험 강화

---

### 2.3 일상적 사용 단계

**단계:** 아침 확인 (Daily Check)
- **행동:** 로그인 후 할 일 목록 확인
- **생각:** "오늘 뭐 해야 하지? 빨간색으로 표시된 건 뭐지?"
- **감정:** 집중, 약간의 긴장감 (기한 경과 확인 시)
- **터치포인트:** 할 일 목록, 기한 경과 섹션
- **기회:** 명확한 시각적 구분으로 우선순위 즉시 파악

**단계:** 새로운 할 일 추가
- **행동:** 미팅 중 떠오른 작업을 스마트폰으로 빠르게 추가
- **생각:** "모바일에서도 추가가 쉽네. 나중에 자세히 수정하면 되겠다."
- **감정:** 편리함, 안도감
- **터치포인트:** 모바일 할 일 추가 폼
- **기회:** 모바일 최적화로 이동 중 빠른 입력 지원

**단계:** 완료 처리
- **행동:** 작업 완료 후 체크박스 클릭
- **생각:** "체크 표시하니까 성취감이 드네. 완료 목록으로 내려가는구나."
- **감정:** 성취감, 만족감
- **터치포인트:** 체크박스, 완료 섹션
- **기회:** 즉각적인 시각적 피드백과 완료 목록 분리로 동기부여

**단계:** 마감일 조정
- **행동:** 일정 변경으로 할 일 수정, 마감일 연장
- **생각:** "수정이 간단하네. 클릭 몇 번이면 끝이네."
- **감정:** 통제감, 편안함
- **터치포인트:** 할 일 수정 폼
- **기회:** 빠른 수정 기능으로 유연성 제공

---

### 2.4 장기 사용 단계

**단계:** 습관화 (1-2주 후)
- **행동:** 매일 아침 로그인, 저녁 정리가 루틴이 됨
- **생각:** "이제 이것 없이는 일정 관리가 안 될 것 같아."
- **감정:** 신뢰, 의존성
- **터치포인트:** 전체 서비스
- **기회:** 지속적인 가치 제공으로 장기 사용자 전환

**단계:** 완료 목록 확인
- **행동:** 주말에 이번 주 완료한 할 일 목록 확인
- **생각:** "이번 주 꽤 많이 했네. 뿌듯하다."
- **감정:** 성취감, 자부심
- **터치포인트:** 완료된 할 일 섹션
- **기회:** 완료 목록으로 성취감 강화, 지속 사용 동기 부여

**단계:** 추천 및 공유
- **행동:** 지인에게 서비스 추천
- **생각:** "이거 진짜 간단하고 좋아. XX한테 알려줘야지."
- **감정:** 만족, 자신감
- **터치포인트:** 소셜 공유, 구전
- **기회:** 만족한 사용자가 자연스럽게 홍보

## 3. 상세 사용자 시나리오

### 3.1 회원가입 및 온보딩

#### 시나리오 A: 첫 방문 사용자의 회원가입

**배경:**
김미영은 프리랜서 디자이너로, 여러 프로젝트 마감일 관리가 어려워 새로운 할 일 관리 도구를 찾던 중 지인 추천으로 서비스를 알게 되었다.

**목표:**
간단하게 가입하고 바로 사용 시작하기

**시나리오:**

1. **서비스 접속**
   - 미영은 브라우저에서 서비스 URL 입력
   - 로그인 화면이 표시되고 "회원가입" 링크 확인
   - "회원가입" 링크 클릭

2. **회원가입 페이지 진입**
   - 이메일과 비밀번호 입력 필드만 있는 간단한 폼 확인
   - "복잡한 정보 요구 안 하네, 좋아" 생각하며 긍정적 반응

3. **정보 입력**
   - 이메일: `mikyung.kim@gmail.com` 입력
   - 비밀번호: `Design2026!` 입력 (12자)
   - 비밀번호 재확인 필드에 동일하게 입력

4. **유효성 검증**
   - 이메일 형식 자동 검증
   - 비밀번호 길이 검증 (최소 8자)
   - 모든 필드가 유효하면 가입 버튼 활성화

5. **가입 완료**
   - "가입하기" 버튼 클릭
   - "회원가입이 완료되었습니다" 메시지 표시
   - 자동으로 로그인 페이지로 리다이렉트

6. **첫 로그인**
   - 방금 입력한 이메일과 비밀번호로 로그인
   - JWT 토큰 발급 및 로컬스토리지 저장
   - 할 일 목록 페이지로 이동

**결과:**
- 미영은 2분 이내에 가입 및 로그인 완료
- 복잡한 절차 없이 바로 사용 시작 가능
- 긍정적인 첫인상으로 계속 사용할 의지 생김

**Given-When-Then:**
```
Given 미영이 회원가입 페이지에 접속했을 때
When 유효한 이메일(mikyung.kim@gmail.com)과 8자 이상 비밀번호를 입력하고
And "가입하기" 버튼을 클릭하면
Then 회원가입이 성공적으로 완료되고
And 로그인 페이지로 자동 리다이렉트되며
And "회원가입이 완료되었습니다" 메시지가 표시된다
```

---

#### 시나리오 B: 중복 이메일로 가입 시도

**배경:**
박준호는 이전에 가입했던 사실을 잊고 동일한 이메일로 재가입을 시도한다.

**목표:**
중복 이메일 감지 및 명확한 안내

**시나리오:**

1. **회원가입 페이지 진입**
   - 준호는 회원가입 페이지 접속
   - 이메일: `junho.park@email.com` 입력 (이미 등록된 이메일)
   - 비밀번호: `Student123` 입력

2. **가입 시도**
   - "가입하기" 버튼 클릭
   - 서버에서 이메일 중복 검사 수행

3. **오류 처리**
   - 400 Bad Request 응답 수신
   - "이미 사용 중인 이메일입니다" 메시지 표시
   - 이메일 입력 필드 하단에 빨간색 오류 메시지

4. **문제 해결**
   - 준호는 이미 가입했던 것을 기억
   - "로그인" 링크 클릭하여 로그인 페이지로 이동
   - 기존 계정으로 로그인 성공

**결과:**
- 명확한 오류 메시지로 문제 즉시 인지
- 로그인 페이지 링크로 쉽게 해결 방법 찾음
- 중복 계정 생성 방지

**Given-When-Then:**

   - 화면 최상단에 빨간색 "기한 경과" 섹션 표시
   - "팀 회의록 정리" 1건이 어제 마감이었음을 인지
   - "어제 깜빡했네, 오늘 꼭 해야겠다" 긴장감

2. **우선순위 최상위 인식**
   - 기한 경과 할 일이 모든 할 일보다 위에 위치
   - 빨간색 시각적 강조로 긴급성 명확히 인지
   - 오늘 마감인 다른 할 일보다 먼저 처리하기로 결정

3. **즉시 작업 착수**
   - 다른 작업 미루고 회의록 정리 시작
   - 1시간 후 작업 완료

4. **완료 처리**
   - 할 일 목록으로 돌아와 체크박스 클릭
   - 기한 경과 섹션에서 완료 섹션으로 이동
   - 빨간색 강조 사라짐
   - "휴, 이제 오늘 일에 집중하자" 안도감

5. **교훈**
   - "마감일 지난 건 빨간색으로 바로 보이니까 놓칠 일이 없겠네"
   - 앞으로 마감일 관리 더 철저히 하겠다고 다짐

**결과:**
- 기한 경과 할 일을 명확히 인지
- 시각적 강조로 긴급성 전달
- 즉시 처리하도록 유도

**Given-When-Then:**
```
Given 성은이 할 일 목록을 확인할 때
When 마감일이 지났고 미완료인 할 일이 있으면
Then 화면 최상단 "기한 경과" 섹션에 빨간색으로 표시되고
And 다른 모든 할 일보다 위에 위치하며
When 해당 할 일을 완료 처리하면
Then 기한 경과 섹션에서 사라지고
And 완료 섹션으로 이동한다
```

#### 시나리오 C: 마감일 변경 (일정 조정)

**배경:**

변경된 마감일 업데이트

   - 스마트폰으로 할 일 목록 접속

2. **해당 할 일 찾기**
   - "오늘 할 일" 섹션에서 "자료구조 과제 2번" 발견
   - 현재 마감일: 이번 주 금요일 (2일 후)

3. **수정 모드 진입**
   - 할 일 카드에서 편집 버튼(연필 아이콘) 탭
   - 수정 폼 표시

4. **마감일 변경**
   - 마감일 선택기 열기
   - 다음 주 월요일 날짜 선택
   - 제목과 설명은 변경하지 않음
   - "저장" 버튼 탭

5. **결과 확인**
   - 수정된 할 일이 "오늘 할 일"에서 "예정된 할 일"로 이동
   - 마감일 순 재정렬로 더 아래 위치
   - "이제 여유 있게 할 수 있겠다" 안도감

6. **다른 할 일 우선순위 조정**
   - 금요일까지 여유가 생겨 다른 과제에 집중 가능
   - 우선순위 자연스럽게 재조정

**결과:**
- 간단한 수정으로 마감일 업데이트
- 자동 재정렬로 우선순위 반영
- 유연한 일정 관리 가능

**Given-When-Then:**
```
Given 준호가 "자료구조 과제 2번" 할 일을 수정 모드로 열었을 때
When 마감일을 금요일에서 다음 주 월요일로 변경하고
And "저장" 버튼을 클릭하면
Then 마감일이 업데이트되고
And 할 일이 "오늘 할 일"에서 "예정된 할 일"로 이동하며

```

### 3.5 완료된 할 일 관리

#### 시나리오 A: 완료 처리 및 즉시 확인

**배경:**
김미영은 오후 3시, "클라이언트 A 로고 1차 시안"을 완성하고 납품했다. 할 일을 완료 처리하려 한다.

**목표:**
작업 완료 처리 및 성취감 확인

**시나리오:**

1. **작업 완료**
   - 미영은 시안을 이메일로 전송 완료
   - 할 일 목록 페이지 열기
   - "클라이언트 A 로고 1차 시안" 할 일 찾기

2. **완료 체크**
   - 해당 할 일의 체크박스 클릭
   - 즉각적인 시각적 변화:
     - 제목에 취소선 표시
     - 전체 카드가 회색으로 변경
     - 0.3초 애니메이션과 함께 아래로 이동

3. **완료 섹션 이동**
   - 할 일이 미완료 목록에서 사라짐
   - "완료된 할 일 (5)" 섹션으로 이동
   - 섹션 카운트가 4에서 5로 증가

4. **성취감 확인**
   - "오늘 벌써 5개 완료했네!" 뿌듯함
   - 완료 섹션 펼쳐서 오늘 한 일들 확인
   - 동기부여되어 다음 작업 착수

5. **미완료 목록 확인**
   - 완료 처리 후 남은 할 일 3건 확인
   - "조금만 더 하면 오늘 할 일 다 끝나겠네" 동기부여

**결과:**
- 즉각적인 완료 처리로 성취감
- 시각적 피드백으로 긍정적 경험
- 완료 목록으로 진행 상황 추적

**Given-When-Then:**
```
Given 미영이 "클라이언트 A 로고 1차 시안" 작업을 완료했을 때
When 해당 할 일의 체크박스를 클릭하면
Then 제목에 취소선이 표시되고
And 카드가 회색으로 변경되며
And 미완료 목록에서 "완료된 할 일" 섹션으로 이동하고
And 완료 카운트가 1 증가한다
```

#### 시나리오 B: 완료 목록 조회 (주간 성과 확인)

**배경:**

**목표:**
주간 완료 목록 조회 및 성과 확인

**시나리오:**

   - "완료된 할 일 (23)" 섹션 발견

2. **완료 섹션 펼치기**
   - "완료된 할 일 (23)" 클릭하여 펼치기
   - 이번 주 완료한 23건의 할 일 목록 표시
   - 각 항목에 완료 날짜(updated_at) 표시

3. **성과 확인**
   - 월요일: 3건 완료

4. **뿌듯함과 동기부여**

   - "다음 주도 이렇게 해야지" 동기부여

5. **완료 섹션 닫기**
   - 확인 완료 후 섹션 다시 접기
   - 미완료 목록에만 집중
   - 주말 동안 휴식

**결과:**
- 주간 성과를 시각적으로 확인
- 완료 목록으로 성취감 및 동기부여
- 자신의 생산성 패턴 파악

**Given-When-Then:**
```
Given 성은이 금요일 저녁 할 일 목록에 접속했을 때
When "완료된 할 일 (23)" 섹션을 펼치면
Then 이번 주 완료한 23건의 할 일이 표시되고
And 각 항목에 완료 날짜가 표시되며
And 주간 성과를 한눈에 확인할 수 있고
When 섹션을 다시 클릭하면
Then 완료 목록이 접혀 미완료 목록에만 집중할 수 있다
```

### 3.6 할 일 정리 및 삭제

#### 시나리오 A: 불필요한 할 일 삭제

**배경:**
김미영은 클라이언트가 프로젝트를 취소하여 관련 할 일을 삭제하려 한다.

불필요해진 할 일 제거

**시나리오:**

   - 할 일 목록에서 정리하기로 결정

2. **첫 번째 할 일 삭제**
   - "클라이언트 D 로고 시안 1차" 할 일 카드에서 삭제 버튼(휴지통 아이콘) 클릭

   - "확인" 버튼 클릭
   - 할 일이 목록에서 즉시 사라짐
   - 0.3초 fade-out 애니메이션

4. **나머지 할 일 삭제**
   - 동일한 방법으로 관련 할 일 2건 추가 삭제
   - 각각 확인 대화상자로 신중하게 삭제

5. **결과 확인**
   - 총 3건 삭제 완료
   - 미완료 목록이 깔끔하게 정리됨
   - "이제 현재 프로젝트에만 집중할 수 있겠네" 만족감

**결과:**
- 불필요한 할 일 쉽게 제거
- 확인 대화상자로 실수 방지
- 깔끔한 목록 유지

**Given-When-Then:**
```
Given 미영이 "클라이언트 D 로고 시안 1차" 할 일을 삭제하려 할 때
When 삭제 버튼을 클릭하면
Then "정말 삭제하시겠습니까?" 확인 대화상자가 표시되고
When "확인" 버튼을 클릭하면
Then 할 일이 데이터베이스에서 영구 삭제되고
And 목록에서 즉시 사라진다
When "취소" 버튼을 클릭하면
Then 삭제가 취소되고 할 일이 유지된다
```

   - "너무 많아서 스크롤이 길어지네, 정리해야겠다"

2. **개별 삭제 진행**
   - 2주 이상 지난 완료 할 일부터 삭제 시작
   - 각 할 일의 삭제 버튼 클릭
   - 확인 대화상자에서 "확인" 선택
   - 한 건씩 삭제 (10건 삭제)

3. **시간 소요 인지**
   - "하나씩 삭제하니까 시간이 좀 걸리네"
   - "하지만 실수로 지울까봐 확인하는 게 나은 것 같기도 하고..."
   - 10건 삭제 후 멈춤

4. **필요한 것만 남김**
   - 최근 2주 완료 목록은 유지하기로 결정
   - "나중에 참고할 수도 있으니까"
   - 정리 완료

5. **향후 기능 희망**
   - "나중에 일괄 삭제나 자동 보관 기능 있으면 좋겠다"
   - (Phase 2 기능으로 피드백)

**결과:**
- 개별 삭제로 신중한 정리
- 완료 목록 관리로 깔끔한 인터페이스
- 향후 개선 사항 피드백 발생

**Given-When-Then:**
```
Given 준호가 완료 목록에서 오래된 할 일을 삭제하려 할 때
When 각 할 일의 삭제 버튼을 클릭하고
And 확인 대화상자에서 "확인"을 선택하면
Then 해당 완료 할 일이 영구 삭제되고
And 완료 목록 카운트가 감소하며
And 깔끔한 목록이 유지된다
```

## 4. 엣지 케이스 및 예외 시나리오

### 4.1 잘못된 입력

#### 시나리오 A: 잘못된 이메일 형식으로 가입 시도

**배경:**
신규 사용자가 이메일 형식을 잘못 입력한다.

**시나리오:**

**Given:** 회원가입 페이지에 접속
**When:** 이메일 필드에 "invalidemail" (@ 없음) 입력
**Then:**
- "유효한 이메일 주소를 입력하세요" 오류 메시지 표시
- 가입 버튼 비활성화
- 이메일 필드에 빨간색 테두리

**When:** 이메일 필드에 "user@" (도메인 없음) 입력
**Then:**
- 동일한 오류 메시지
- 가입 진행 불가

**When:** 올바른 형식 "user@example.com" 입력
**Then:**
- 오류 메시지 사라짐
- 테두리 정상 색상 복원
- 가입 버튼 활성화

---

#### 시나리오 B: 비밀번호 길이 부족

사용자가 짧은 비밀번호를 입력한다.

**시나리오:**

**Given:** 회원가입 페이지
**When:** 비밀번호 필드에 "123" (3자) 입력
**Then:**
- "비밀번호는 최소 8자 이상이어야 합니다" 메시지 표시
- 가입 버튼 비활성화

**When:** "1234567" (7자) 입력
**Then:**
- 여전히 오류 메시지 표시

**When:** "12345678" (8자) 입력
**Then:**
- 오류 메시지 사라짐
- 가입 버튼 활성화

---

#### 시나리오 C: 제목 200자 초과 입력

**배경:**
사용자가 매우 긴 제목을 입력한다.

**시나리오:**

**Given:** 할 일 추가 폼
**When:** 제목 필드에 300자 텍스트 붙여넣기
**Then:**
- 입력 필드가 200자까지만 수용
- 또는 "제목은 최대 200자까지 입력 가능합니다" 메시지
- 초과 문자는 자동 잘림 또는 입력 차단

---

### 4.2 네트워크 오류

#### 시나리오 A: 로그인 중 네트워크 끊김

**배경:**
사용자가 로그인 중 인터넷 연결이 끊긴다.

**시나리오:**

**Given:** 로그인 페이지에서 이메일/비밀번호 입력 완료
**When:** "로그인" 버튼 클릭
**And:** 네트워크 연결 끊김
**Then:**
- 로딩 인디케이터 표시
- 5초 타임아웃 후 "네트워크 오류가 발생했습니다. 인터넷 연결을 확인해주세요" 메시지
- 로그인 버튼 다시 활성화

**When:** 네트워크 복구 후 다시 로그인 시도
**Then:**
- 정상적으로 로그인 진행

---

#### 시나리오 B: 할 일 추가 중 서버 오류

**배경:**
서버가 일시적으로 응답하지 않는다.

**시나리오:**

**Given:** 할 일 추가 폼에 모든 정보 입력 완료
**When:** "추가" 버튼 클릭
**And:** 서버가 500 Internal Server Error 반환
**Then:**
- "일시적인 오류가 발생했습니다. 잠시 후 다시 시도해주세요" 메시지
- 입력한 데이터는 폼에 유지 (사용자가 다시 입력할 필요 없음)
- "다시 시도" 버튼 표시

**When:** "다시 시도" 버튼 클릭
**And:** 서버 정상 복구
**Then:**
- 할 일 생성 성공
- 목록에 추가됨

---

#### 시나리오 C: 완료 처리 중 네트워크 지연

**배경:**
사용자가 체크박스를 클릭했지만 네트워크가 느려 응답이 지연된다.

**시나리오:**

**Given:** 할 일 목록 페이지
**When:** 체크박스 클릭 (완료 처리)
**And:** 네트워크 응답 3초 지연
**Then:**
- 즉시 UI에서 시각적 피드백 (취소선, 회색 처리)
- 로딩 인디케이터 (체크박스 옆)
- 3초 후 서버 응답 수신하면 로딩 인디케이터 사라짐

**If:** 서버 응답이 실패하면
**Then:**
- UI 상태 원래대로 복원 (체크 해제)
- "완료 처리 중 오류가 발생했습니다" 메시지
- 사용자가 다시 시도 가능

---

### 4.3 권한 없는 접근 시도

#### 시나리오 A: 로그인하지 않은 상태에서 할 일 페이지 접근

**배경:**
인증되지 않은 사용자가 URL로 직접 접근을 시도한다.

**시나리오:**

**Given:** 로그인하지 않은 상태
**When:** 브라우저에서 `/todos` URL 직접 입력
**Then:**
- 즉시 로그인 페이지로 리다이렉트
- "로그인이 필요한 서비스입니다" 메시지 표시

**When:** 로그인 성공
**Then:**
- 원래 접근하려던 `/todos` 페이지로 자동 이동

---

#### 시나리오 B: 다른 사용자의 할 일 수정 시도 (API 직접 호출)

**배경:**
악의적인 사용자가 다른 사용자의 할 일 ID로 API를 직접 호출한다.

**시나리오:**

**Given:** 사용자 A(ID: 1)로 로그인
**When:** 사용자 B(ID: 2)의 할 일(todo_id: 100, user_id: 2) 수정 API 호출
```
PUT /api/todos/100
Authorization: Bearer {user_A_token}
{ "title": "해킹 시도" }
```
**Then:**
- 서버에서 소유권 검증
- user_id 불일치 감지
- 403 Forbidden 응답
- "접근 권한이 없습니다" 오류 메시지

**And:**
- 사용자 B의 할 일은 변경되지 않음
- 보안 로그 기록 (선택사항)

---

#### 시나리오 C: 만료된 JWT 토큰으로 API 요청

**배경:**
사용자의 토큰이 만료되었지만 계속 API를 호출한다.

**시나리오:**

**Given:** 로그인 후 7일 경과 (JWT 토큰 만료)
**When:** 할 일 목록 조회 API 호출
```
GET /api/todos
Authorization: Bearer {expired_token}
```
**Then:**
- 서버에서 JWT 검증 실패
- 401 Unauthorized 응답
- 프론트엔드에서 401 감지
- 자동으로 로그아웃 처리
- 로그인 페이지로 리다이렉트
- "세션이 만료되었습니다. 다시 로그인해주세요" 메시지

---

### 4.4 동시 수정 시도

#### 시나리오 A: 여러 디바이스에서 동시 수정

**배경:**
사용자가 데스크탑과 모바일 모두에서 로그인하여 같은 할 일을 동시에 수정한다.

**시나리오:**

**Given:** 데스크탑과 모바일 양쪽에서 로그인
**And:** 같은 할 일(ID: 50) 화면에 표시 중
**When:**
- 데스크탑에서 제목을 "A"로 수정하고 저장 (시간: 10:00:00)
- 모바일에서 제목을 "B"로 수정하고 저장 (시간: 10:00:03)
**Then:**
- 각 요청이 순차적으로 서버에 도달
- 데스크탑 수정: updated_at = 10:00:00, title = "A"
- 모바일 수정: updated_at = 10:00:03, title = "B"
- 최종 DB 상태: title = "B" (마지막 수정이 덮어씀)

**When:** 데스크탑 화면 새로고침
**Then:**
- 최신 데이터("B") 로딩
- 데스크탑의 수정("A")은 덮어써짐

**참고:**
- MVP에서는 "Last Write Wins" 방식
- 향후 버전에서 충돌 감지 및 병합 기능 고려

---

#### 시나리오 B: 삭제된 할 일 수정 시도

**배경:**
사용자가 한 디바이스에서 할 일을 삭제했는데, 다른 디바이스에서 수정을 시도한다.

**시나리오:**

**Given:** 데스크탑과 모바일 양쪽에서 로그인
**And:** 할 일(ID: 60) 양쪽에 표시 중
**When:**
- 데스크탑에서 해당 할 일 삭제
- 모바일(삭제 사실 모름)에서 해당 할 일 수정 시도
**Then:**
- 모바일에서 PUT /api/todos/60 요청
- 서버에서 404 Not Found 응답
- "할 일을 찾을 수 없습니다" 오류 메시지
- 모바일 화면에서 해당 할 일 자동 제거
- 사용자에게 "다른 디바이스에서 삭제된 할 일입니다" 안내

---

### 4.5 데이터 경계값 테스트

#### 시나리오 A: 마감일이 없는 할 일

**배경:**
사용자가 마감일 없이 할 일을 생성한다.

**시나리오:**

**Given:** 할 일 추가 폼

**And:** 할 일 생성
**Then:**

- 목록에서 "마감일 없음" 표시
- 기한 경과 섹션에 표시 안 됨
- 마감일 있는 할 일보다 아래 정렬

---

#### 시나리오 B: 과거 날짜를 마감일로 설정

**배경:**
사용자가 이미 지난 날짜를 마감일로 설정한다.

**시나리오:**

**Given:** 할 일 추가 폼

**참고:**

#### 시나리오 C: 매우 먼 미래 날짜 설정

**And:** 할 일 생성
**Then:**
- 정상적으로 생성
- "예정된 할 일" 섹션에 표시
- 마감일 순 정렬로 가장 아래 위치
- 특별한 제한 없음

## 5. 사용 빈도 및 시나리오 우선순위

### 5.1 High Priority (매일 여러 번 사용)

**빈도:** 하루 5-10회 이상

1. **할 일 목록 조회**
   - 사용 빈도: 최고
   - 시간대: 아침, 점심, 저녁, 틈날 때마다
   - 디바이스: 모바일 > 데스크탑
   - 중요도: 최우선 최적화 필요

2. **새로운 할 일 추가**
   - 사용 빈도: 하루 3-5회
   - 시간대: 업무 중, 회의 후, 생각날 때마다
   - 디바이스: 모바일 ≈ 데스크탑
   - 중요도: 빠른 입력 중요

3. **완료 처리 (체크박스 클릭)**
   - 사용 빈도: 하루 3-7회
   - 시간대: 작업 완료 즉시
   - 디바이스: 데스크탑 > 모바일
   - 중요도: 즉각적인 피드백 필요

4. **기한 경과 할 일 확인**
   - 사용 빈도: 하루 2-3회
   - 시간대: 아침, 점심 후
   - 디바이스: 모바일 ≈ 데스크탑
   - 중요도: 시각적 강조 필수

---

### 5.2 Medium Priority (주 1-2회 사용)

**빈도:** 주간 2-5회

5. **할 일 수정 (제목, 설명, 마감일)**
   - 사용 빈도: 주 3-4회
   - 시간대: 일정 변경 시
   - 디바이스: 데스크탑 > 모바일
   - 중요도: 편리한 UI 필요

6. **할 일 삭제**
   - 사용 빈도: 주 2-3회
   - 시간대: 불필요해진 작업 정리 시
   - 디바이스: 데스크탑 ≈ 모바일
   - 중요도: 확인 단계 필수

7. **완료 목록 조회**
   - 사용 빈도: 주 1-2회
   - 시간대: 주말, 금요일 저녁
   - 디바이스: 데스크탑 > 모바일
   - 중요도: 성취감 제공 중요

8. **마감일 변경**
   - 사용 빈도: 주 2-3회
   - 시간대: 일정 조정 필요 시
   - 디바이스: 데스크탑 > 모바일
   - 중요도: 빠른 수정 지원

---

### 5.3 Low Priority (월 1회 이하)

**빈도:** 월간 1-2회

9. **회원가입**
   - 사용 빈도: 1회 (최초)
   - 시간대: 서비스 발견 시
   - 디바이스: 데스크탑 ≈ 모바일
   - 중요도: 첫인상 중요, 간단해야 함

10. **로그인 (세션 만료 후)**
    - 사용 빈도: 월 1회 미만 (자동 로그인 유지)
    - 시간대: 토큰 만료 시
    - 디바이스: 모바일 ≈ 데스크탑
    - 중요도: 빠르고 편리해야 함

11. **로그아웃**
    - 사용 빈도: 월 1회 미만
    - 시간대: 디바이스 변경 시, 보안 필요 시
    - 디바이스: 공용 컴퓨터
    - 중요도: 명확한 위치 필요

12. **완료 목록 대량 정리**
    - 사용 빈도: 월 1회
    - 시간대: 월말, 분기말
    - 디바이스: 데스크탑
    - 중요도: 개선 필요 (향후 일괄 삭제 기능)

---

### 5.4 우선순위 매트릭스

| 기능 | 빈도 | 중요도 | 최적화 우선순위 |
|------|------|--------|-----------------|
| 할 일 목록 조회 | 최고 | 최고 | P0 |
| 완료 처리 | 높음 | 높음 | P0 |
| 할 일 추가 | 높음 | 최고 | P0 |
| 기한 경과 확인 | 높음 | 높음 | P0 |
| 할 일 수정 | 중간 | 중간 | P1 |
| 할 일 삭제 | 중간 | 중간 | P1 |
| 마감일 변경 | 중간 | 중간 | P1 |
| 완료 목록 조회 | 낮음 | 중간 | P2 |
| 회원가입 | 최저 | 높음 (첫인상) | P1 |
| 로그인 | 낮음 | 중간 | P2 |
| 로그아웃 | 최저 | 낮음 | P3 |

**P0:** 성능 및 UX 최우선 최적화
**P1:** 중요도 높음, 안정성 확보
**P2:** 기본 기능 제공
**P3:** 필수 기능이지만 빈도 낮음

## 6. 모바일 vs 데스크탑 사용 시나리오

### 6.1 모바일 우선 시나리오

#### 시나리오 1: 출퇴근 중 할 일 확인

**상황:**
지하철에서 스마트폰으로 오늘 할 일 빠르게 확인

**디바이스 선택 이유:**
- 이동 중이라 데스크탑 사용 불가
- 빠른 확인만 필요
- 한 손으로 조작 가능

**모바일 최적화 요구사항:**
- 빠른 로딩 (2초 이내)
- 한 손 조작 가능한 UI
- 중요 정보 상단 배치 (기한 경과, 오늘 할 일)
- 최소 스크롤로 주요 정보 파악

**사용 패턴:**
1. 앱 접속 (자동 로그인)
2. 상단 기한 경과 섹션 확인 (빨간색)
3. 오늘 할 일 섹션 확인 (주황색)
4. 3-5초 내 확인 완료
5. 앱 종료

---

#### 시나리오 2: 회의 중 빠른 할 일 추가

**상황:**
회의실에서 팀장이 지시한 작업을 즉시 기록

**디바이스 선택 이유:**
- 노트북이 없거나 꺼내기 불편한 상황
- 5초 내 빠르게 추가 필요
- 회의 흐름 방해하지 않아야 함

**모바일 최적화 요구사항:**
- 제목만으로 빠른 추가 가능
- 자동완성 키보드 최적화
- 마감일 선택 간편화 (오늘, 내일, 다음 주 버튼)
- 추가 후 즉시 폼 초기화

**사용 패턴:**
1. 스마트폰 잠금 해제
2. 앱 접속
3. 제목 필드 탭
4. "클라이언트 보고서 작성" 입력
5. 마감일 "내일" 버튼 탭
6. "추가" 버튼 탭
7. 5-10초 내 완료

---

#### 시나리오 3: 외출 중 할 일 완료 처리

**상황:**
외부에서 전화로 업무 완료 후 스마트폰으로 체크

**디바이스 선택 이유:**
- 사무실 밖이라 데스크탑 접근 불가
- 완료 기록 즉시 필요
- 잊어버리기 전에 처리

**모바일 최적화 요구사항:**
- 큰 터치 타겟 (체크박스 최소 44x44px)
- 즉각적인 시각적 피드백
- 실수로 다른 항목 체크 방지
- 완료 취소 쉽게 가능

**사용 패턴:**
1. 앱 접속
2. 해당 할 일 찾기
3. 체크박스 탭
4. 취소선 및 회색 처리 확인
5. 완료 섹션 이동 확인
6. 성취감 느끼며 앱 종료

---

### 6.2 데스크탑 우선 시나리오

#### 시나리오 4: 업무 시작 전 상세 계획 수립

**상황:**
월요일 아침 사무실에서 이번 주 일정 상세 계획

**디바이스 선택 이유:**
- 큰 화면으로 전체 목록 한눈에 파악
- 여러 할 일 동시 비교 가능
- 키보드로 빠른 입력
- 드래그 앤 드롭 등 정교한 조작 (향후 기능)

**데스크탑 최적화 요구사항:**
- 넓은 화면 활용
- 여러 섹션 동시 표시
- 키보드 단축키 지원 (Enter로 추가 등)
- 상세 정보 입력 편의성

**사용 패턴:**
1. 데스크탑에서 로그인
2. 전체 할 일 목록 스크롤하며 확인
3. 이번 주 마감일 할 일 5건 발견
4. 새로운 할 일 3건 추가 (상세 설명 포함)
5. 기존 할 일 2건 마감일 조정
6. 우선순위 판단 후 업무 시작

---

#### 시나리오 5: 할 일 상세 수정 및 정리

**상황:**
저녁 시간 사무실에서 할 일 상세 정보 보완 및 정리

**디바이스 선택 이유:**
- 긴 설명 입력에는 키보드가 편리
- 여러 항목 비교하며 수정
- 대량 정리 작업
- 집중력 필요한 작업

**데스크탑 최적화 요구사항:**
- 텍스트 에디터 같은 편안한 입력 환경
- 복사/붙여넣기 편의성
- 여러 항목 빠르게 이동
- 마우스/키보드 조합 사용

**사용 패턴:**
1. 아침에 모바일로 추가한 할 일 찾기
2. 수정 모드로 상세 설명 추가
3. 이메일에서 요구사항 복사하여 붙여넣기
4. 마감일 정확히 설정
5. 불필요한 할 일 3건 삭제
6. 완료 목록 정리

---

#### 시나리오 6: 주간 성과 리뷰

**상황:**
금요일 저녁 이번 주 완료 목록 확인하며 주간 리뷰

**디바이스 선택 이유:**
- 많은 완료 목록 스크롤하며 확인
- 큰 화면으로 전체 성과 파악
- 프린트 또는 스크린샷 가능 (향후)
- 여유 있게 확인

**데스크탑 최적화 요구사항:**
- 완료 목록 필터링/정렬 (향후 기능)
- 날짜별 그룹핑 (향후 기능)
- 스크롤 성능
- 통계 시각화 (향후 기능)

**사용 패턴:**
1. 완료된 할 일 섹션 펼치기
2. 이번 주 완료 목록 스크롤
3. 주요 성과 회상
4. 다음 주 계획 수립
5. 만족감과 함께 퇴근

## 7. 시나리오 기반 테스트 체크리스트

- [ ] 중복 이메일 가입 차단
- [ ] 잘못된 이메일 형식 차단
- [ ] 짧은 비밀번호 차단
- [ ] 가입 후 로그인 페이지 리다이렉트
- [ ] 올바른 자격증명으로 로그인 성공
- [ ] 잘못된 비밀번호 로그인 실패
- [ ] 로그인 후 토큰 저장 및 자동 로그인
- [ ] 로그아웃 후 토큰 삭제

### 7.2 할 일 생성 시나리오 테스트

- [ ] 제목만으로 간단히 추가
- [ ] 제목 + 설명 + 마감일 완전한 추가
- [ ] 제목 없이 추가 시도 차단
- [ ] 생성 후 목록에 즉시 표시
- [ ] 입력 폼 자동 초기화
- [ ] 생성 시 Incomplete 상태 자동 설정
- [ ] 마감일 없는 할 일 생성
- [ ] 과거 마감일로 생성 시 기한 경과 표시

### 7.3 할 일 조회 및 정렬 시나리오 테스트

- [ ] 자신의 할 일만 조회
- [ ] 기한 경과 할 일 최상단 표시
- [ ] 오늘 마감 할 일 강조 표시
- [ ] 마감일 순 정렬
- [ ] 마감일 없는 할 일 하단 배치
- [ ] 완료 할 일 분리 표시
- [ ] 빈 목록 시 안내 메시지

### 7.4 할 일 수정 시나리오 테스트

- [ ] 제목 수정 성공
- [ ] 설명 추가/수정 성공
- [ ] 마감일 변경 후 재정렬
- [ ] 수정 취소 시 원래 값 유지
- [ ] 수정 시 updated_at 자동 업데이트
- [ ] 다른 사용자 할 일 수정 차단 (403)

### 7.5 할 일 삭제 시나리오 테스트

- [ ] 삭제 확인 대화상자 표시
- [ ] 확인 시 삭제 성공
- [ ] 취소 시 삭제 취소
- [ ] 삭제 후 목록에서 제거
- [ ] 다른 사용자 할 일 삭제 차단 (403)
- [ ] 존재하지 않는 할 일 삭제 시도 (404)

### 7.6 완료 처리 시나리오 테스트

- [ ] 체크박스 클릭으로 완료 전환
- [ ] 완료 시 취소선 및 회색 표시
- [ ] 완료 섹션으로 이동
- [ ] 완료 취소로 미완료 복원
- [ ] 완료 할 일은 기한 경과 아님
- [ ] 완료 섹션 접기/펼치기

### 7.7 기한 경과 관리 시나리오 테스트

- [ ] 과거 마감일 + Incomplete = Overdue
- [ ] 오늘 마감일 = Today
- [ ] 미래 마감일 = Upcoming
- [ ] 마감일 없음 = Overdue 아님
- [ ] 완료 할 일 = Overdue 아님
- [ ] 기한 경과 빨간색 강조
- [ ] 마감일 변경 시 상태 재계산

- [ ] 제목 200자 초과 차단
- [ ] 설명 1000자 초과 차단

### 7.9 모바일 시나리오 테스트

- [ ] 모바일 화면(320px)에서 정상 표시
- [ ] 터치 타겟 44x44px 이상
- [ ] 한 손 조작 가능
- [ ] 빠른 추가 (5초 내)
- [ ] 모바일 키보드 최적화
- [ ] 스크롤 성능

## 8. 결론 및 향후 개선 사항

### 8.1 핵심 사용자 니즈 요약

본 문서를 통해 파악한 사용자들의 핵심 니즈:

1. **명확한 우선순위 파악**
   - 기한 경과, 오늘 마감, 예정 할 일을 시각적으로 명확히 구분
   - 자동 정렬로 긴급한 작업 우선 표시

2. **빠른 입력과 접근**
   - 모바일에서 5-10초 내 할 일 추가
   - 제목만으로 간단히 추가 가능
   - 자동 로그인으로 즉시 접근

3. **성취감과 동기부여**
   - 완료 처리의 즉각적인 시각적 피드백
   - 완료 목록으로 주간 성과 확인
   - 취소선과 색상 변화로 진행 상황 인지

4. **유연한 일정 관리**
   - 마감일 쉽게 변경 가능
   - 완료 취소로 실수 복구
   - 삭제 확인으로 신중한 결정

5. **디바이스 간 원활한 전환**
   - 모바일에서 빠른 추가, 데스크탑에서 상세 작성
   - 자동 동기화
   - 반응형 UI로 일관된 경험

### 8.2 MVP 이후 개선 방향

사용자 시나리오 분석을 통해 도출된 향후 개선 사항:

**Phase 2 (출시 후 1개월):**
- 일괄 삭제 기능 (완료 목록 대량 정리)

- 마감일 빠른 선택 (오늘, 내일, 다음 주 버튼)
- 완료 목록 날짜별 그룹핑

**Phase 3 (출시 후 3개월):**
- 검색 기능
- 필터링 (마감일, 상태별)
- 카테고리/태그 (업무/개인 구분)
- 통계 대시보드 (주간 완료율 등)

**Phase 4 (출시 후 6개월):**
- 알림 및 리마인더

### 8.3 성공 지표 (KPI)

사용자 시나리오 기반 성공 지표:

**사용성:**
- 첫 할 일 추가까지 소요 시간: 평균 2분 이내
- 일일 활성 사용자(DAU): 70% 이상
- 주간 활성 사용자(WAU): 90% 이상
- 평균 세션 시간: 3-5분

**효율성:**
- 모바일 할 일 추가 시간: 평균 10초 이내
- 페이지 로딩 시간: 2초 이내
- 완료 처리 응답 시간: 1초 이내

**만족도:**
- 7일 사용자 유지율: 60% 이상
- 30일 사용자 유지율: 40% 이상
- 추천 의향(NPS): 40 이상
- 사용자 피드백 긍정률: 80% 이상
